Matematiksel ifadeleri expression tree olarak göstermek, o gösterim evaluate etmek: 1 + 3 / ((2 + 1) * 4³)

Bu expression tree yapısını in-order traverse ederek sonucu bulabiliriz. Her seferinde karşımıza bu kadar basit gramer yapıları çıkacak değil. Yazılım dillerinde matematiksel ifadelerden başka yapılar da mevcut, örneğin aşağıdaki kod parçasını ağaca oturtmak istersek,

if (6 == 6) {
var1 = 7 + func();
}

Aslında epey kolay! Metin formundaki kodu bir şekilde ağaç yapısına dönüştürdüğümüzde değer hesaplamak için tree traversal yeterli oluyor! Peki, bu ağacı nasıl oluşturacağız? Asıl soruna geliyoruz: lexing ve sonrasında parsing*.

Lexical Analysis

String* üzerinde çalışmak genel olarak zor. Ağaca oturtma sürecinde bir sayının değeri, değişkenin ya da fonksiyonun ismi bizi ilgilendirmiyor. Tokenizer*, metin işleme sürecinde syntax elemanlarının anlamını çıkarırız. Mesela, if (x == 12) b = 21;i ele alalım.

lexeme token seminfo
if IF
( LPAREN
x IDENT "x"
== EQUALS
12 NUM 12
) RPAREN
b IDENT "b"
= ASGN
21 NUM 21
; SEMI

Genelde, bütün metni tutmak yerine gereksiz detayları atlayarak ham token bilgisini tutmayı tercih ederiz. token isminde bir enum tanımlayıp onu seminfo union tanımlayarak gramer yapılarını daha özlü bir şekilde temsil etmek mümkündür:

enum token {
TK_IF,
TK_LPAREN,
// ...
};
union seminfo {
char *id;
size_t num;
}
